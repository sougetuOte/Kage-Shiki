# 仕様・実装同期ルール

## 目的

Wave 4 監査で発見された仕様漏れパターン（trigram tokenizer 未記載、Protocol 外メソッド未文書化等）の再発を防止する。

## ルール

### S-1: 実装変更時の仕様突合

コードに新しい定数・アルゴリズム・データ構造・メソッドを追加したとき、
対応する `docs/specs/` のセクションを**再読**し、記載があるか確認する。
記載がなければ同サイクル内で仕様を更新する。

**チェック対象例**:
- SQL スキーマの変更（tokenizer、カラム、インデックス）
- Protocol / Interface へのメソッド追加・変更
- 定数値・Enum の追加
- エラーメッセージの新規追加

### S-2: Protocol 外メソッドの明示

Protocol / Interface に定義されていないメソッドを具象クラスに追加する場合、
仕様書に「Protocol 外メソッドとして提供」と明記する。
呼び出し側が Protocol ではなく具象型に依存する必要があることを文書化する。

### S-3: 仕様の「Phase 2 対応」マーク

実装しない仕様項目がある場合、仕様書に「Phase 2: ...を検討」と明記する。
仕様書に書いてあるが実装しないという暗黙のスキップを禁止する。

### S-4: Refactor 後の仕様再読

Refactor ステップ（TDD の Green → Refactor）でコード構造を変更した場合、
変更がインターフェースや公開 API に影響するかを確認し、
影響がある場合は仕様を同期更新する。

## 適用タイミング

- BUILDING フェーズ: TDD サイクルの Green 直後（R-1 と同時に実施）
- AUDITING フェーズ: 仕様・実装ズレの監査項目として検証
