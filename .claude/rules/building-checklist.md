# BUILDING 品質チェックリスト

Phase 1 監査・Retro で発見された不具合パターンを防止するルール。
TDD サイクルの各ステップで適用する。

## Red（テスト作成時）

- **R-4: FR チェックリスト駆動テスト**
  タスク定義の FR 番号一覧をテストファイル docstring に転記する。
  各 FR に最低 1 テストが対応することを確認してから実装に進む。

- **R-5: 異常系テストの義務**
  正常系だけでなく、エラーパス・境界値・例外ケースのテストも Red で書く。
  「try/except があるなら、except に落ちるテストがある」を原則とする。

## Green（実装時）

- **R-2: 有限セットは dict ディスパッチ**
  3 分岐以上の有限値セット（purpose, state, type 等）は if-chain ではなく dict/map で処理する。
  未登録値は KeyError で明示的に失敗させる。

- **R-6: else のデフォルト値禁止**
  if-elif-else や dict.get() のデフォルト値を書く際、
  「この else/default に到達する正当な入力」を言語化できなければ `raise ValueError` にする。

- **R-3: 定数定義 → 使用の即時接続**
  定数・frozenset・Enum を定義したら、同じサイクル内でその定数を参照するコードを書く。
  「後で使う予定」の未参照定数を残さない。

## Green 直後（実装完了時）

- **R-11: Green 直後 3 点ミニチェック**（Phase 1 Retro Try-1 由来）
  Green 完了時に必ず以下 3 点を確認する（`/full-review` まで待たない）:
  (a) **カプセル化**: コンストラクタ引数で注入すべきものをグローバル参照していないか
  (b) **R-6 再確認**: else/default に到達する正当な入力を言語化できるか
  (c) **S-1 仕様突合**: 追加した定数・メソッドが仕様書に記載されているか

- **R-1: FR 突合チェック**
  テスト通過後、対応する FR/設計仕様のセクションを**再読**し、
  フィールド名・定数値・メッセージ文言が**文字単位で一致する**か照合する。

- **R-5 続: カバレッジ確認**
  `pytest --cov` で未カバー行を確認する。
  未カバーが「テスト不要（GUI イベント等）」か「テスト必要（エラーパス等）」かを判断し、
  必要なら追加テストを書く。目標: 各モジュール 90% 以上。

- **S-1: 仕様同期チェック**（`spec-sync.md` 参照）
  新しい定数・メソッド・スキーマを追加した場合、`docs/specs/` の対応セクションに
  記載があるか確認する。記載がなければ同サイクル内で仕様を更新する。

- **R-7: スレッド安全性チェック**（L-1 由来）
  マルチスレッドで共有されるオブジェクト（DB接続、キュー、状態変数等）を扱う場合、
  スレッド間の受け渡しパスをテストで検証する。
  SQLite は `check_same_thread=False` 等のスレッド制約を設計時に明示すること。

- **R-8: 永続状態の「2回目起動」テスト**（L-2 由来）
  DB やファイルに永続化する機能は、「既存データがある状態での起動・実行」を
  テストケースに含める。`:memory:` だけでなく既存レコードありのシナリオを検証する。

- **R-9: シャットダウン経路の一意性チェック**（L-3 由来）
  シャットダウン処理が複数経路（トレイ終了、Ctrl+C、atexit 等）から呼ばれる場合、
  全経路で「1回だけ実行」が保証されることを検証するテストを書く。

- **R-10: GUI 変更時の目視確認**（L-5 由来）
  GUI のレイアウト・ウィジェットを変更した場合、実際に起動して以下を目視確認する:
  - 長文テキストの表示（スクロール・はみ出し）
  - ウィンドウリサイズ時のレイアウト崩れ
  - 操作フロー（入力→送信→表示）の動作確認
