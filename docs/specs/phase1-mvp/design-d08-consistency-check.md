# D-8: 整合性チェック3類型の具体的検出方針

**決定対象**: requirements.md Section 8 — D-8「整合性チェック3類型の具体的検出方針: プロンプト指示 or ルールベース、FR-6.4 の実装詳細」
**関連 FR**: FR-6.3, FR-6.4, FR-6.5
**ステータス**: 提案（承認待ち）

---

## 1. コンテキスト

FR-6.4 は「consistency_interval メッセージごとに軽量な整合性チェックを実行する」と定める。02-personality-system.md では PCL（Personality Consistency Learning）から引用した3類型を監視対象として挙げている。

| 類型 | 説明 |
|------|------|
| キャラクター幻覚 | 現実の知識がキャラクター設定外から漏出する（「ChatGPT として...」等） |
| 行動記述の曖昧性 | 行動の意図が不明確になる（感情が読めない、なぜその行動を取るか不明） |
| 一般知識忘却 | 役割特化が過ぎて汎用能力が低下する（「わたしは AI なので数学ができません」等） |

Phase 1 の制約は「追加 API コールなし」（システムプロンプト内指示方式）である（requirements.md Section 9 Mediator 結論）。したがって、別途 LLM を呼び出して整合性を検査するのではなく、通常の応答生成時のシステムプロンプトに「自己確認指示」を埋め込む方式を採用する。

設計の核心は「指示文面の具体性」にある。曖昧な指示は LLM に素通りされる。各類型に対応する **検出観点** と **確認タイミング** を明確に定義することが本設計の目的である。

---

## 2. 選択肢分析

### 選択肢 A: 単一の汎用整合性確認指示（シンプル方式）

**概要**: 「人格と一貫しているか確認してから回答せよ」という1文をシステムプロンプトに追加する。

**メリット**:
- 実装が最もシンプル
- トークン消費が最小

**デメリット**:
- 指示が抽象的すぎて LLM が実際に確認しない可能性が高い
- 3類型のどれを確認すべきかが不明確
- 「確認した」というコンプライアンスのみが発生し、実質的な効果が出ない

### 選択肢 B: 3類型に対応した具体的な自己問答指示（構造化方式）

**概要**: 3類型それぞれに対応する具体的な確認観点をシステムプロンプトに埋め込む。`consistency_check_active` フラグが True の時のみ表示される（D-3 の応答規範ブロックに統合）。

**メリット**:
- 各類型に対応した観点が明確で、LLM が実際に確認しやすい
- 確認結果を出力しない指示と組み合わせることで、ユーザー体験を損なわない
- D-3 の設計と整合する（consistency_check_active フラグの活用）

**デメリット**:
- 指示文が長くなるため、consistency_interval 間隔での一時的なトークン増加が発生する（推定 150-250 トークン）
- 指示文の内容が複雑になると、LLM が「確認」を前置きとして出力しやすくなる

### 選択肢 C: ルールベースによる事前検出 + LLM 補完

**概要**: 応答テキストに特定のパターン（「ChatGPT」「私は AI です」「その質問には答えられません」等）が含まれている場合に、再生成をトリガーする後処理を設ける。

**メリット**:
- ルールベース部分は確実に検出できる（偽陰性がない）
- LLM の内部確認に依存しない

**デメリット**:
- 追加 API コールが発生する（再生成が必要な場合）
- Phase 1 の制約（追加 API コールなし）と矛盾する可能性がある
- ルールで検出できるのは表面的なパターンのみ。深層的な一貫性崩れは検出できない

---

## 3. Three Agents Perspective

**[Affirmative]**: 選択肢 B は Phase 1 の制約（追加 API コールなし）を守りながら、3類型それぞれに対する実質的な確認を実現できる。重要なのは「確認結果を出力しないこと」という指示の存在であり、これにより整合性チェックがユーザーの会話体験を邪魔しない。consistency_interval（デフォルト 15 メッセージ）ごとにしか発火しないため、長期的なトークンコストへの影響も限定的である。

**[Critical]**: 選択肢 B においても、「確認結果を出力しないこと」という指示への LLM の遵守は保証されない。特に Haiku のような小型モデルでは、自己確認の思考過程が応答に漏れ出る可能性がある。また、指示が 3 項目にわたることで「確認が複雑だから省略しよう」という LLM の暗黙の最適化が起きるリスクがある。選択肢 C のルールベース部分は Phase 1 でも実装すべきで、追加 API コールなし（再生成なし）でもログ記録だけ行う軽量版として共存させると良い。

**[Mediator]**: 選択肢 B を主体とし、選択肢 C の軽量版（再生成なし、ログ記録のみ）を補完的に組み合わせる複合方式を採用する。「確認結果を出力しないこと」という指示については、D-3 の応答規範ブロックに明示することで LLM の遵守率を高める。ルールベース後処理はフラグとしてログに記録するのみとし、Phase 2 以降でのパターン分析に活用する。

---

## 4. 決定

**採用**: 選択肢 B（3類型対応の構造化自己問答指示）+ 選択肢 C 軽量版（再生成なし、ログ記録のみ）の複合方式

**理由**:
- Phase 1 の「追加 API コールなし」制約を遵守
- 3類型それぞれに対応した具体的な確認指示により、汎用指示よりも実効性が高い
- ルールベース後処理（ログのみ）により、キャラクター幻覚の代表的なパターンを確実に記録し、将来の改善に活用できる

---

## 5. 詳細仕様

### 5.1 整合性チェックの発火タイミング

```
message_count = 0   # セッション開始時
message_count += 1  # 各 user メッセージ受信ごとにインクリメント

consistency_check_active = (message_count % consistency_interval == 0)
```

`consistency_interval` のデフォルト: 15（config.toml `[memory].consistency_interval`）

**セッション開始時のカウント**: セッション開始メッセージ（AgentCore 起動時の挨拶）は message_count に含めない。ユーザーの実際の発言のみをカウントする。

---

### 5.2 類型別の検出観点と確認指示文面

#### 類型1: キャラクター幻覚（Character Hallucination）

**定義**: LLM が「自分は AI である」「ChatGPT/Claude/Gemini などのモデルである」という自己認識を露出させること。または、ペルソナと無関係な「私は〜として作られました」等の説明的発言。

**根本原因**: LLM の基底モデルが持つ自己同一性がキャラクターペルソナを「突き破る」現象。

**確認指示文面**:

```
1. 今回の応答において、「私は AI です」「私はアシスタントです」「私は〇〇モデルです」
   という表現、または <persona> に定義された名前・一人称以外のアイデンティティを
   示す表現が含まれていないかを確認すること。
   もし含まれている場合は、<persona> に基づいたキャラクターとして言い直すこと。
```

**ルールベース後処理パターン（ログ記録用）**:

```python
CHARACTER_HALLUCINATION_PATTERNS = [
    r"私はAIです",
    r"私はアシスタントです",
    r"私は言語モデルです",
    r"ChatGPT",
    r"Claude",
    r"Gemini",
    r"anthropic",
    r"OpenAI",
    r"私はプログラムされた",
    r"AIとして",
    r"言語モデルとして",
]
```

マッチした場合: `WARNING` レベルでログに記録し、`consistency_check_hit_type_1` フラグを立てる。Phase 1 では再生成はしない。

---

#### 類型2: 行動記述の曖昧性（Action Ambiguity）

**定義**: 応答から感情や行動の意図が読み取れなくなること。具体的には以下のパターン:
- 文体が突然ニュートラルになる（「了解しました。それについて説明します。」等）
- 感情表現が消えて事務的なトーンになる
- 口調パターン（C6）と一致しない丁寧語や専門用語が混入する

**確認指示文面**:

```
2. 今回の応答の文体・語尾・語彙が <style_samples> の参照例および
   <persona> の「C6: 口調パターン」と一致しているかを確認すること。
   特に、事務的・説明的な文体（「〜について説明します」「了解しました」等）に
   なっていないかを確認すること。
```

**ルールベース後処理パターン（ログ記録用）**:

```python
ACTION_AMBIGUITY_PATTERNS = [
    r"了解しました",
    r"承知しました",
    r"ご質問にお答えします",
    r"について説明します",
    r"以下にまとめます",
    r"ご要望に応じて",
    r"お役に立てれば",
]
```

**注意**: このパターンマッチは誤検知が多い。あくまでログ記録用であり、キャラクターによっては丁寧語が正常な口調の場合もある。pattern hit があった場合は persona_core.md の C6 を参照してコンテキストで判断する（Phase 2 以降の自動判断に向けた記録）。

---

#### 類型3: 一般知識忘却（General Knowledge Degradation）

**定義**: キャラクターへの特化が過ぎて、通常の会話能力や知識が失われること。具体的には:
- 「私はそのような質問には答えられません」という回避（本来は可能な質問）
- 一般的な知識質問に対して「わからない」を連発する
- C11（知識の自己認識）で「知らないことは素直に言う」が設定されているキャラクターが、知っていることまで「知らない」と言う

**確認指示文面**:

```
3. 今回の応答で「答えられません」「わかりません」という表現を使う場合、
   <persona> の「C11: 知識の自己認識」に照らして適切かを確認すること。
   キャラクターとして本当に知らない情報であれば正直に言ってよいが、
   一般的な知識についてまで「私には答えられない」と言わないこと。
```

**ルールベース後処理パターン（ログ記録用）**:

```python
KNOWLEDGE_DEGRADATION_PATTERNS = [
    r"答えられません",
    r"回答できません",
    r"その情報は持っていません",
    r"私の能力を超えています",
    r"対応できかねます",
]
```

### 5.3 完全な自己確認指示ブロック（D-3 との統合）

D-3 の `[S7] 応答規範ブロック` 内の `{{#if consistency_check_active}}` セクションに埋め込む完全な指示文は以下の通りである。

```
【本ターンの自己確認】
今回の応答において、以下の3点を自己確認してから返答すること。確認結果は出力しないこと:

1. アイデンティティの確認: 「私はAIです」「私はアシスタントです」等の表現、
   または <persona> に定義された名前・一人称以外のアイデンティティを示す表現が
   含まれていないか確認すること。含まれている場合は <persona> に基づき言い直すこと。

2. 口調の確認: 文体・語尾・語彙が <style_samples> の参照例および <persona> の
   「C6: 口調パターン」と一致しているか確認すること。事務的・説明的な文体に
   なっていないか確認すること。

3. 知識応答の確認: 「答えられません」「わかりません」という表現を使う場合、
   <persona> の「C11: 知識の自己認識」に照らして適切か確認すること。
   一般的な知識についてまで回避的にならないこと。
```

この指示ブロックのトークン数: 約 200-250 トークン。

---

### 5.4 ルールベース後処理の実装位置

ルールベース後処理は AgentCore の応答後処理フェーズで実行する。処理フローは以下の通りである。

```
LLM 応答受信
    │
    ▼
display_text() でユーザーに表示  ← 表示は即座に行う
    │
    ▼
[非同期後処理]
  ルールベースパターンマッチング（全パターン）
  ヒット有り → WARNING ログ記録
  ヒット有り → consistency_hit カウンタをインクリメント（セッション中の累計）
    │
    ▼
observations に書込
human_block 更新判断
personality_trends 提案判断
```

後処理は表示と非同期で行うため、ユーザーの応答速度には影響しない。

---

### 5.5 ログ記録の形式

整合性チェックのヒットは以下の形式でログに記録する。

```
WARNING consistency_check: type=1 (character_hallucination), session_id=xxxx, message_count=30, pattern="ChatGPT"
WARNING consistency_check: type=2 (action_ambiguity), session_id=xxxx, message_count=45, pattern="承知しました"
WARNING consistency_check: type=3 (knowledge_degradation), session_id=xxxx, message_count=60, pattern="答えられません"
```

Phase 1 ではこのログは開発者向けの監視情報として使用する。Phase 4（04-unified-design.md のロードマップ）での「整合性チェックの精度向上」フェーズで、このログデータを元にパターンの調整と検出閾値の最適化を行う。

---

### 5.6 consistency_interval の調整方針

| 値 | 挙動 | 適用場面 |
|----|------|---------|
| 5 | 高頻度チェック（living-agents の ReflectionEngine と同等）。トークンコスト増大 | デバッグ・初期テスト |
| 15（デフォルト） | 会話が自然に流れつつ、30分〜1時間程度の会話セッションで1-2回発火 | 通常運用 |
| 30 | 長時間会話でのみ発火。整合性崩れへの反応が遅くなるが、コストは最小 | 軽量運用・低スペック環境 |
| 0 | 整合性チェック無効化（テスト用） | 開発時のみ |

`consistency_interval = 0` の場合は `consistency_check_active` を常に False にする（ゼロ除算の回避）。

---

### 5.7 Phase 2 以降への拡張パス

Phase 1 では「追加 API コールなし」の制約から、整合性チェックは自己確認指示の埋め込みとルールベース後処理のみで実装する。Phase 2 以降では以下の拡張を検討する。

| フェーズ | 拡張内容 |
|---------|---------|
| Phase 2 | ルールベースの検出パターンを会話ログから機械学習して更新（教師なし） |
| Phase 3 | utility モデルを用いた軽量な整合性スコアリング（毎回ではなく類型2の疑いが高い時のみ） |
| Phase 4 | Phase 1 から蓄積した consistency_hit ログを用いた検出閾値の最適化と、パターンライブラリの拡充 |

## 6. 影響範囲

| 影響先 | 内容 |
|--------|------|
| D-3（プロンプトテンプレート） | 自己確認指示ブロックを応答規範ブロック内の `consistency_check_active` フラグ制御で挿入する（既に D-3 に反映済み） |
| AgentCore | consistency_check_active フラグの計算（message_count % consistency_interval）と応答後処理のルールベースパターンマッチング |
| config.toml | `[memory].consistency_interval` の値がチェック頻度を制御する |
| ログシステム（D-2） | consistency_check WARNING ログのフォーマットと出力先を D-2 の設計と整合させる |
| Phase 2+ | ルールベースの検出ログが Phase 4 の整合性チェック精度向上の入力データとなる |
