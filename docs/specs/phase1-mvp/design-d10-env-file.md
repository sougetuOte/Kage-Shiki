# D-10: .env ファイルによる環境変数設定

**決定対象**: requirements.md Section 8 D-10「.env ファイルによる環境変数設定 — python-dotenv 採用可否」
**関連 FR**: FR-1.6（起動時に ANTHROPIC_API_KEY 環境変数の存在を確認する）、NFR-8（API キーは環境変数 ANTHROPIC_API_KEY を使用）、NFR-3（外部依存を最小化する）
**ステータス**: 提案（承認待ち）

---

## 1. コンテキスト

影式の API キー管理について、要件は以下のとおりである。

| 要件 | 内容 |
|------|------|
| NFR-8 | API キーは config.toml に保存しない。環境変数 `ANTHROPIC_API_KEY` を使用する |
| NFR-3 | 外部依存を最小化する（tkinter, pystray, anthropic のみ） |
| FR-1.6 | 起動時に `ANTHROPIC_API_KEY` 環境変数の存在を確認する。未設定時は設定方法を案内してアプリを終了する |

ターゲットユーザーは非開発者の可能性が高い。Windows の環境変数設定（システムのプロパティ → 環境変数）は、一般ユーザーには馴染みが薄く、UX 上の摩擦が大きい。

`.env` ファイルによる設定は、プロジェクトディレクトリに `.env` を置くだけで環境変数を設定できる利便性がある一方、外部ライブラリ依存やセキュリティリスクの観点で検討が必要である。

配置先（D-1 参照）: 処理は `main.py` の最初期、または `core/config.py` の初期化時に実行する。

---

## 2. 選択肢分析

### 選択肢 A: python-dotenv 採用

`python-dotenv` ライブラリを使用して `.env` ファイルを自動ロードする。

```toml
# pyproject.toml（依存関係追加のイメージ。実装コードではない）
dependencies = [
    "pystray",
    "anthropic",
    "python-dotenv",   # 追加
]
```

`.env` ファイルの例:
```
ANTHROPIC_API_KEY=sk-ant-api03-...
```

起動時の処理イメージ（擬似コード）:
```
# main.py の先頭で dotenv を読み込む
dotenv.load_dotenv()
# 以降 os.environ["ANTHROPIC_API_KEY"] が有効になる
```

- **概要**: デファクトスタンダードのライブラリ。`load_dotenv()` 一行で `.env` を読み込む
- **メリット**: 初回ユーザーが `.env` ファイルを作成するだけで API キーを設定できる。システム環境変数の操作が不要。開発環境でも本番環境と分離してキーを管理できる。GitHub での誤コミット防止（`.gitignore` に `.env` を追加する慣例が広く知られている）
- **デメリット**: NFR-3 が「tkinter, pystray, anthropic のみ」と明示した外部依存に追加ライブラリが加わる。`python-dotenv` 自体は軽量だが、原則を破ることになる

### 選択肢 B: os.environ のみ（.env 不採用）

`python-dotenv` を使用せず、OS の環境変数設定のみに依存する。`.env` ファイルは読み込まない。

```
# Windows での設定方法（ユーザーが行う操作）
1. システムのプロパティ → 環境変数
2. ユーザー環境変数に ANTHROPIC_API_KEY を追加
または
3. コマンドプロンプト: set ANTHROPIC_API_KEY=sk-ant-...
   （セッション限定、永続化されない）
または
4. PowerShell: $env:ANTHROPIC_API_KEY = "sk-ant-..."
   （セッション限定）
```

- **概要**: 追加ライブラリなしで NFR-3 を完全に遵守する
- **メリット**: 外部依存が増えない。OS レベルの環境変数は全プロセスで共有され、影式以外の用途にも使える
- **デメリット**: Windows の環境変数 GUI は非開発者には難しい。システム環境変数への設定はアプリ再起動が必要（ユーザーが混乱しやすい）。コマンドプロンプトでの設定はセッション限定で永続化されない。初回セットアップの摩擦が大きく US-1（日常的にテキスト対話したい）の達成を阻害しうる

### 選択肢 C: 自前 .env パーサー実装（外部依存なし）

標準ライブラリのみで簡易な `.env` ファイルパーサーを実装する。

`.env` ファイルの想定形式（行ごとに `KEY=VALUE`）:
```
ANTHROPIC_API_KEY=sk-ant-api03-...
# コメント行
```

パーサーの実装イメージ（擬似コード）:
```
# core/config.py 内の load_env_file(path) 関数のイメージ
# 処理内容:
#   - .env ファイルを行単位で読み込む
#   - '#' で始まる行はコメントとしてスキップ
#   - '=' で分割して KEY / VALUE を取得
#   - os.environ に設定（既存の環境変数は上書きしない）
#   - ファイルが存在しない場合はスキップ（エラーにしない）
```

- **概要**: `python-dotenv` の核心機能（KEY=VALUE 形式の読み込み）のみを実装する
- **メリット**: 外部依存を増やさずに `.env` ファイルのサポートを実現できる。NFR-3 を完全に遵守。実装は 30 行程度で済む
- **デメリット**: クォート処理（`KEY="value with space"`）、エスケープ（`KEY=value\nwith\nnewline`）等のエッジケースを自前で処理する必要がある。`python-dotenv` が対応している複雑な書式を全てサポートするには相当の工数がかかる。「標準的でない実装」はユーザーが予期しない動作をする可能性がある

### 選択肢 D: README でのガイド強化（.env 非対応 + 丁寧な案内）

`.env` ファイル機能は提供せず、代わりに初回セットアップを丁寧に案内する。

- FR-1.6 の未設定時のエラーメッセージを詳細化する
- README に Windows 環境変数設定の画像付きガイドを用意する
- 未設定時のエラーメッセージで設定手順の URL を案内する

```
# エラーメッセージの例（擬似コード）
未設定時の表示:

  ANTHROPIC_API_KEY が設定されていません。

  設定方法（Windows）:
  1. スタートメニュー → 「環境変数」で検索 →「システム環境変数の編集」
  2. 「環境変数」ボタン → ユーザー環境変数の「新規」
  3. 変数名: ANTHROPIC_API_KEY
     変数値: [Anthropic コンソールで取得したキー]
  4. OK を押して、影式を再起動

  詳細は README をご覧ください。
```

- **概要**: ライブラリ追加も自前実装もせず、ユーザー教育で解決する
- **メリット**: 実装コストゼロ。外部依存なし。OS 標準の環境変数を習得することで、ユーザーのリテラシーが上がる
- **デメリット**: 初回セットアップの摩擦が最も高い。US-2（プレビューで確認してから確定）のような「試しやすさ」とは逆方向のアプローチ。Windows の環境変数 GUI はアプリ再起動が必要で、エラーメッセージを見ながら設定できない

## 3. Three Agents Perspective

**[Affirmative]**（推進者の視点）

自前 .env パーサー実装（選択肢 C）が最もバランスが良い。

python-dotenv（選択肢 A）の「外部依存を1つ追加する」コストに対して、得られる恩恵はファイル読み込みの30行程度のコードである。影式が NFR-3 で「tkinter, pystray, anthropic のみ」と明示した理由は、インストール摩擦の排除と依存関係管理の単純化である。python-dotenv 自体は軽量・安定したライブラリだが、原則を一度破れば次の追加が容易になるという slippery slope リスクがある。

自前実装の場合、対応すべき基本書式（`KEY=VALUE`、コメント行、既存環境変数の保持）を明確に定義すれば、エッジケースの数は限られる。ユーザーが `.env` ファイルに書くのは `ANTHROPIC_API_KEY=sk-ant-...` の1行だけであり、クォートやエスケープが必要なシナリオは現実的に発生しない。

**[Critical]**（批判者の視点）

自前実装（選択肢 C）のリスクは「なまじ動くので問題に気づきにくい」点である。例えば Windows の CRLF（`\r\n`）改行、BOM（UTF-8 with BOM）、キーの前後の空白（`KEY = VALUE`）等、ユーザーが普通のテキストエディタで編集すると発生しうる書式の揺れが実装を複雑にする。

また「インストール手順書に `.env` の作り方を書く」選択肢 D と比較して、自前実装がユーザーに与える実質的な UX 改善効果は限定的かもしれない。選択肢 D + 丁寧なドキュメントで FR-1.6 の要件は十分に満たせる。

python-dotenv（選択肢 A）については、NFR-3 の「tkinter, pystray, anthropic のみ」という記述は「それ以外は一切禁止」ではなく「最小化する」の文脈で読むべきであり、ユーザー体験の観点で十分な根拠があれば追加できると解釈できる。

**[Mediator]**（調停者の結論）

NFR-3 の「最小化する」は「ゼロにする」ではない。追加の正当化基準は以下とする:

1. **セキュリティクリティカルか**: API キーの読み込みは誤りが許されない処理である → Yes
2. **自前実装のバグコスト**: CRLF/BOM/クォート/空白等のエッジケースで API キーが正しく読めないバグは、デバッグ困難かつユーザー影響が直接的
3. **ライブラリの品質**: python-dotenv は PyPI 月間 3000万+ DL、pure Python、transitive dependency ゼロ、長期メンテナンス実績あり

python-dotenv（選択肢 A）を採用する。NFR-3 の依存リストに python-dotenv を追加する。

自前の30行パーサー（選択肢 C）は一見シンプルだが、「誤ってはならない処理を自前で書くリスク」が「依存を1つ追加するリスク」を上回る。slippery slope の懸念は、追加基準（セキュリティクリティカル + battle-tested + transitive dependency ゼロ）を明示することで防ぐ。

---

## 4. 決定

**採用**: 選択肢 A — python-dotenv 採用

**理由**:
1. **セキュリティクリティカル**: API キーの読み込みは誤りが許されない。battle-tested のライブラリに委ねることでバグリスクを排除する
2. **UX 改善**: `.env` ファイルを `load_dotenv()` 一行で読み込み、Windows 環境変数 GUI の操作を不要にする
3. **エッジケース網羅**: クォート、BOM、CRLF、空白、エスケープ等を python-dotenv が全て処理する。自前実装ではこれらの各ケースがバグの温床になる
4. **NFR-3 更新**: 依存リストに python-dotenv を追加する。追加基準: セキュリティクリティカル + pure Python + transitive dependency ゼロ + 月間 3000万+ DL の実績

## 5. 詳細仕様

### 5.1 .env ファイルの仕様

**ファイルパス**: アプリケーションのルートディレクトリ（`main.py` が存在するディレクトリの親 = パッケージルート）からの相対パス。`Path(".")` ではなく `Path(__file__).resolve().parent.parent / ".env"` 相当の絶対パス解決を使用する

**サポートする書式**:

```
# .env.example（ユーザー向けテンプレート）
# Anthropic API キーを設定してください
# https://console.anthropic.com/ で取得できます
ANTHROPIC_API_KEY=sk-ant-api03-ここにキーを貼り付ける
```

**書式ルール**:

| ルール | 内容 |
|--------|------|
| 行形式 | `KEY=VALUE` 形式（= の前後にスペースがあっても許容） |
| コメント | `#` で始まる行は無視 |
| 空行 | 無視 |
| 文字エンコーディング | UTF-8（BOM あり/なし 両対応） |
| 改行コード | CRLF（Windows）/ LF 両対応 |
| 既存環境変数 | OS で設定済みの変数は上書きしない（OS 環境変数が優先） |
| クォート | 非対応（`KEY="value"` は `"value"` がそのまま値になる） |
| エスケープ | 非対応 |
| KEY の命名 | `ANTHROPIC_API_KEY` のみ想定（他のキーも読み込まれるが、現在使用するものは1つ） |

### 5.2 python-dotenv の使用仕様

```
# main.py の先頭（他の import より前）で呼び出す
from dotenv import load_dotenv
load_dotenv(override=False)  # OS 環境変数を上書きしない

# override=False により:
#   - .env に ANTHROPIC_API_KEY があり、OS にも設定済み → OS の値が優先
#   - .env に ANTHROPIC_API_KEY があり、OS に未設定 → .env の値が使用される
#   - .env が存在しない → エラーにならない（スキップ）
```

**python-dotenv がサポートする書式**（自前実装では全てカバーが必要だった）:
- `KEY=VALUE`（基本形）
- `KEY="value with spaces"`（クォート対応）
- `KEY='value'`（シングルクォート対応）
- `# コメント行`
- `export KEY=VALUE`（export プレフィックス対応）
- 複数行値（クォート内の改行）
- UTF-8 / BOM 対応

### 5.3 main.py での呼び出しタイミング

起動シーケンスにおける `.env` 読み込みの位置（requirements.md Section 5.5 より抜粋・拡張）:

```
[main.py 起動シーケンス]
  0. .env ファイルの読み込み  ← 新規追加（最初に実行）
     load_dotenv_file(Path(".env"))
  1. config.toml 読み込み
  2. ANTHROPIC_API_KEY 存在確認（FR-1.6）
     ↑ .env の内容が反映された状態でチェック
  3. data_dir 初期化
  ...
```

ステップ 0 はステップ 2（API キー確認）より前に実行する必要がある。

### 5.4 FR-1.6 との連携（API キー未設定時のメッセージ）

`.env` ファイル導入後、未設定時のエラーメッセージを段階的に案内する。

```
# エラーメッセージの構造（擬似コード。実際のテキストは D-6 で決定）

[段階1] 環境変数未設定を検出
[段階2] .env ファイルが存在するか確認
  - .env がある場合: 「.env ファイルは見つかりましたが ANTHROPIC_API_KEY が見つかりません」
  - .env がない場合: 「ANTHROPIC_API_KEY が設定されていません」

[共通] 設定方法の案内:
  方法1（推奨）: プロジェクトフォルダの .env.example を .env にコピーして
                 ANTHROPIC_API_KEY に API キーを貼り付けてください

  方法2: Windows の環境変数として設定してください
         [スタートメニュー] → 「環境変数」で検索 → [環境変数を編集]
         変数名: ANTHROPIC_API_KEY
         変数値: [API キー]

  API キーの取得: https://console.anthropic.com/
```

### 5.5 .env.example テンプレート

リポジトリに `.env.example` を含める（`.env` は `.gitignore` に追加）。

```
# 影式 (Kage-Shiki) 環境設定ファイル
#
# このファイルを .env という名前でコピーして、
# ANTHROPIC_API_KEY にあなたの API キーを設定してください。
#
# API キーの取得: https://console.anthropic.com/
# ========================
ANTHROPIC_API_KEY=sk-ant-api03-ここにキーを貼り付けてください
# ========================
#
# 注意: .env ファイルには秘密情報が含まれます。
# Git リポジトリには含めないでください。（.gitignore で除外済み）
```

### 5.6 セキュリティ考慮

| リスク | 対策 |
|--------|------|
| `.env` の誤コミット | `.gitignore` に `.env` を追加（`.env.example` は含める） |
| ログへの API キー漏洩 | D-2 ログポリシーに従い、環境変数の値はログに出力しない（存在確認の結果のみ記録） |
| `.env` の権限 | Phase 1 では対応しない（Phase 4 でファイル暗号化とともに検討） |
