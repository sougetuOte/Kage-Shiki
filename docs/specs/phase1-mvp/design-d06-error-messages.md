# D-6: エラーメッセージ一覧

**決定対象**: requirements.md Section 8 — D-6「エラーメッセージ一覧（ユーザー向けメッセージ定義）」
**関連 FR**: FR-7.1, FR-7.2, FR-7.3, FR-7.4, FR-7.5, FR-1.6, FR-4.8
**ステータス**: 提案（承認待ち）

---

## 1. コンテキスト

### 1.1 なぜこの決定が必要か

影式はデスクトップマスコットであり、エラーメッセージもキャラクターの「声」で届けることができる。しかし人格が未確定の状態（ウィザード中、起動失敗時）ではキャラクターは存在しないため、システムメッセージとして届ける必要がある。

また、エラーメッセージの「トーン」と「表示先」は UX に直接影響する:
- キャラクターの口調で届けることでユーザーの不安を和らげられる
- しかし深刻なエラーを軽い口調で伝えると、問題の重要性が伝わらない
- エラーごとに「ウィンドウ内表示」か「システムトレイ通知」かが異なる（D-9 の決定と連携）

### 1.2 エラーメッセージの分類軸

本仕様では以下の2軸で全エラーを整理する:

**軸1: トーン**
- **キャラクター口調**: 人格確定済みの通常動作中のエラー。マスコット自身が語る形式
- **システムメッセージ**: 人格未確定時（ウィザード中、起動失敗時）のエラー。キャラクターを前提としない

**軸2: 重篤度**
- **Critical**: アプリが続行できない。ユーザーのアクションが必要
- **Warning**: 続行できるが注意が必要
- **Info**: 状況の通知（自動回復済み等）

## 2. 選択肢分析

### 選択肢 A: 全エラーをキャラクター口調で統一

全てのエラーメッセージをマスコットの人格を反映した口調で記述する。人格が未設定の場合は汎用的な口調（「ごめんなさい、...」等）で代替する。

- **メリット**: UX が一貫している。エラーに遭遇してもキャラクターの「存在感」を感じられる
- **デメリット**: 人格未設定時の「キャラクター的な口調」は不自然。ウィザードが起動しない状況でキャラクターが話すのは論理的に矛盾する

### 選択肢 B: 人格確定前後で完全に分ける

- 人格確定前（ウィザード中・起動失敗）: システムメッセージ（中立的・技術的）
- 人格確定後（通常動作中）: キャラクター口調（マスコット自身が語る）

- **メリット**: 論理的整合性が高い。ユーザーが「今は誰が話しているか」を直感的に理解できる
- **デメリット**: 人格確定後のエラーメッセージは人格ごとに異なるべきだが、静的定義では汎用的な表現になる

### 選択肢 C: 選択肢 B + キャラクター口調はテンプレート変数化

選択肢 B を採用した上で、キャラクター口調のメッセージはテンプレート変数（`{name}`、`{user_name}` 等）を使い、実行時に persona_core.md の情報で埋める。

- **メリット**: キャラクターの個性（名前、二人称）を反映できる。汎用的すぎる口調を回避できる
- **デメリット**: テンプレートエンジンが必要（ただし `str.format()` で十分）。メッセージの可読性がやや低下する

## 3. Three Agents Perspective

**[Affirmative]**（推進者）:
選択肢 C が最善。影式の本質は「人格を持つマスコット」であり、エラーメッセージもその人格の一部である。`{name}` で名前を埋めるだけで「このキャラクターがこういう状況に陥っている」という体験が生まれる。実装コストも `str.format()` で最小限。キャラクター口調のメッセージは C4（人格核文）や C6（口調パターン）を反映するのではなく、あくまで汎用的なテンプレートとし、口調の細かな再現はシステムプロンプトの役割に任せる。

**[Critical]**（批判者）:
エラーメッセージのトーンをキャラクター化することには慎重であるべき。深刻なエラー（persona_core.md 読み込み失敗で起動中断）で「ごめんね、うまく起きられなかった...」のような口調は、問題の深刻さをユーザーが過小評価する可能性がある。また、テンプレート変数の埋め込みが失敗した場合（人格読み込みが失敗しているからこそエラーが出ている場合）、`{name}` がそのまま表示される可能性がある。

**[Mediator]**（調停者）:
結論: **選択肢 C を採用するが、重篤度 Critical のメッセージはシステムメッセージ形式を優先する**。

具体的なルール:
1. **Critical エラー**: 人格確定前後を問わず、システムメッセージ形式（中立的・具体的な復旧手順を含む）
2. **Warning エラー（人格確定後）**: キャラクター口調テンプレート（`{name}` 等で埋める）
3. **Info（人格確定後）**: キャラクター口調テンプレート
4. テンプレート変数のフォールバック: 人格読み込みが失敗した場合は変数を `""` で置換し、汎用文言にデグレード

---

## 4. 決定

**採用**: 選択肢 C（人格確定前後で分離 + キャラクター口調はテンプレート変数化）
**ただし**: Critical エラーはシステムメッセージ形式を優先

**理由**:
- 論理的整合性（人格未確定時はキャラクターが存在しない）を保ちつつ、通常動作中の UX を向上させる
- Critical エラーの深刻さを適切に伝えるため、軽い口調との使い分けが必要
- テンプレート変数は `str.format_map()` を使い、欠落キーは安全に空文字で補完する

### 5.2 エラーメッセージ管理方式

#### 5.2.1 メッセージの格納場所

エラーメッセージは実装コード内に定数として定義する（外部ファイルは不使用）。

```
理由:
- Phase 1 では多言語化は対象外（NFR に含まれない）
- メッセージと発生条件のロジックを近くに置くことでメンテナンス性を確保
- 外部ファイル読み込みが失敗した場合のフォールバックが複雑になる
```

#### 5.2.2 テンプレート変数のフォールバック

キャラクター口調のテンプレートに `{name}` 等が含まれる場合、以下の優先順位でフォールバックする:

| 変数 | 主ソース | フォールバック |
|------|---------|-------------|
| `{name}` | persona_core.md C1 | `""` （空文字） |
| `{user_name}` | human_block.md | `""` （空文字） |
| `{invalid_keys}` | config バリデーション結果 | `"不明なキー"` |
| `{persona_path}` | data_dir + `/persona_core.md` | `"不明なパス"` |
| `{error_detail}` | 例外の str(e) | `"詳細不明"` |
| `{missing_fields}` | パース結果 | `"C1（名前）またはC4（人格核文）"` |

`str.format_map(collections.defaultdict(str, **kwargs))` パターンを使用し、未定義キーを空文字に安全変換する。

### 5.3 表示先の判定ロジック（EM-007 詳細）

EM-007（API 認証エラー）は唯一「ウィンドウ表示中かどうか」で表示先が変わる。

```
認証エラー発生
    │
    ├─ MascotView.is_visible() == True
    │       → display_text(EM-007-window-message)
    │
    └─ MascotView.is_visible() == False
            → D-9 の決定に従ってトレイ通知
              D-9 で通知不可と判定された場合:
              → ログのみ記録（次回ウィンドウ表示時に通知）
```

`MascotView.is_visible()` は MascotView Protocol の7番目のメソッドとして追加するか、TkinterMascotView の内部状態として管理するかは実装フェーズで決定する（Protocol 汚染を避けるなら後者）。

### 5.4 ウィンドウ UI における表示形式

#### 5.4.1 専用エラー画面（Critical エラー）

```
┌─────────────────────────────────┐
│  影式 — エラー                   │
│                                  │
│  [エラーメッセージ本文]           │
│                                  │
│  [ログを開く]  [閉じる]          │
└─────────────────────────────────┘
```

- ウィンドウサイズ: 400 x 300（内容によって可変）
- アプリ終了前に最低5秒間表示する（即座に閉じない）

#### 5.4.2 警告バー（Warning エラー、フォールバック済み）

メイン画面またはウィザード画面の最上部に折りたたみ可能なバーを表示:

```
┌─────────────────────────────────┐
│ [!] 設定の一部にデフォルト値を使用中   [×] │
└─────────────────────────────────┘
```

- クリックで詳細を展開表示
- `×` ボタンで閉じる

#### 5.4.3 ウィンドウ内テキスト表示（通常動作中エラー）

`display_text()` を使い、通常の会話テキストと同じエリアにメッセージを表示する。
エラーメッセージと通常発言の見分け方はテキストの内容のみ（特別なマーカーは付与しない）。
