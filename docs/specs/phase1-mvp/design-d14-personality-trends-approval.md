# D-14: personality_trends 承認フローのトリガー手段

**決定対象**: requirements.md Section 8 — D-14「personality_trends 承認フローのトリガー手段: LLM 意図検出 / キーワードマッチ / ボタン UI、FR-4.6 の実装詳細」
**関連 FR**: FR-4.6
**ステータス**: 提案（承認待ち）

---

## 1. コンテキスト

FR-4.6 は以下を定める。

> personality_trends.md を読み込み、AI は変更提案のみ行う（直接書き込み禁止）

受入条件:
> AI の提案がテキストで表示され、ユーザー承認後に追記される

04-unified-design.md は以下の方針を定めている。

> Phase 1: 提案は対話内でテキスト表示 → ユーザーが「承認」と返答 → ファイルに追記
> Phase 3: 専用の承認 UI を検討

この設計では3つの問題を解決する必要がある。

1. **提案トリガー**: AI がいつ personality_trends への提案を行うべきか（何を検出したら提案するか）
2. **提案形式**: 提案テキストをどのような形式で表示するか（ユーザーが承認/却下を明確に判断できる形式）
3. **承認判定**: ユーザーの応答から「承認」か「却下」かをどのように判定するか

personality_trends.md の更新対象は以下の3種類である（04-unified-design.md Section「承認制の仕組み」より）。

| 更新対象 | 説明 | 対応可変層 |
|---------|------|---------|
| 関係性の変化 | 「最近ゲームの話で盛り上がることが多い」等の傾向認識 | L2 |
| 感情の傾向 | 「最近少し打ち解けてきた」等の感情傾向変化 | L5 |
| 新しい口癖候補（supplementary_styles） | 特定フレーズの使用頻度が高まった観測 | — |

## 2. 選択肢分析

### 選択肢 A: LLM 意図検出（完全 LLM 方式）

**概要**: 毎ターンの応答生成後、または一定間隔で、utility モデルを用いて「personality_trends への提案が適切か」を判定し、適切な場合に提案テキストを生成する。

**メリット**:
- 提案タイミングが自然（会話の文脈を正確に読める）
- 提案内容の質が高い（LLM が関係性の変化を深く理解して言語化できる）

**デメリット**:
- 毎ターンまたは定期的な追加 API コールが発生する
- Phase 1 のコスト目安（$5-15/月）を超えるリスクがある
- レイテンシが増加する

### 選択肢 B: ルールベースのトリガー + LLM の提案生成（ハイブリッド方式）

**概要**: ルールベースで「提案すべきタイミング」を検出し、そのタイミングでのみ LLM を用いて提案テキストを生成する。承認判定もルールベース（キーワードマッチ）で行う。

**メリット**:
- 追加 API コールは「提案すべきと判断した時のみ」に限定されるため、コスト予測が立てやすい
- ルールベーストリガーの条件は明確で、デバッグが容易
- 承認判定がシンプル

**デメリット**:
- トリガー条件がルールベースのため、微妙な文脈の変化を見逃す可能性がある
- 提案生成に utility モデルの追加コールが発生する（ただし低頻度）

### 選択肢 C: キーワードマッチのみ（ルールベース完結方式）

**概要**: 提案トリガーも提案内容も、あらかじめ定義したルールとテンプレートで生成する。LLM は提案生成にも使わない。

**メリット**:
- 追加 API コールが一切発生しない
- 実装がシンプルで、予測可能な動作

**デメリット**:
- 提案内容がテンプレート的になり、キャラクターの自然な成長表現から乖離する
- 影式の設計思想（人格の有機的な変化）に合わない

### 選択肢 D: ボタン UI（GUI 方式）

**概要**: ユーザーがボタンを押して明示的に「今の状態を記録して」と指示した時に提案フローを起動する。

**メリット**:
- ユーザーのコントロール感が強い
- 誤提案が起きない

**デメリット**:
- Phase 1 のスコープ外（要件定義書 Section 2.2 で「承認制 UI は Phase 3」と明記）
- ユーザーが自ら操作しない限り personality_trends が更新されない

## 3. Three Agents Perspective

**[Affirmative]**: 選択肢 B は Phase 1 の現実的な制約（コスト・実装複雑度）と、影式の設計思想（人格の有機的な変化）のバランスが最も優れている。ルールベーストリガーで「n 日間でテーマ X が k 回以上登場した」等の客観的条件を定義し、その条件を満たした時のみ LLM で提案文を生成する。これにより API コールは月に数回程度に抑えられる。承認判定の LLM 意図検出は不要で、「承認」「はい」「いいね」等のキーワードマッチで十分である。

**[Critical]**: 選択肢 B のルールベーストリガーには重要な懸念がある。(1) Phase 1 では personality_trends への書き込みのためだけに utility モデルを呼び出すことになり、その API コールのリトライや失敗時の処理が別途必要になる。(2) 承認後の「ファイルへの追記」処理は、追記内容のフォーマットを誰が決めるかという問題がある（LLM が生成した提案テキストをそのまま追記するのか、AgentCore が整形するのか）。(3) 「承認」キーワードマッチは、ユーザーが「それは承認できない」と言った時に誤マッチするリスクがある。キーワード前後の文脈も確認する必要がある。

**[Mediator]**: 選択肢 B を採用する。ただし Critical の指摘を踏まえて以下を調整する。(1) Phase 1 では提案生成に utility モデルの追加コールを使わず、**通常の対話応答内で提案を行う**方式とする（=追加 API コールなし）。AgentCore がトリガー条件を検出したら、次の対話応答の中でキャラクターが自然な流れで提案する。(2) 追記内容のフォーマットはシステムプロンプトで指定し、LLM が所定のフォーマットで提案を生成する。AgentCore は承認後にそのテキストをパースして追記する。(3) 承認判定は「承認」「はい」「OK」等のキーワードマッチとするが、「〜は承認できない」等の否定修飾パターンを除外する。

---

## 4. 決定

**採用**: 選択肢 B の調整版（ルールベーストリガー + 通常対話応答内での提案 + キーワード承認判定）

**理由**:
- Phase 1 の「追加 API コールは最小化」方針に沿う（提案生成を通常対話に統合することで追加コールゼロ）
- ルールベーストリガーにより提案タイミングが予測可能で実装が明確
- Phase 3 の承認 UI への移行パスを保持する

---

## 5. 詳細仕様

### 5.1 提案トリガー条件

AgentCore の応答後処理フェーズで以下の条件を評価する。複数条件が同時に満たされた場合、最も優先度の高い1件のみ提案する（同一セッションで複数提案が重なることを防ぐ）。

#### トリガー T1: 関係性の変化（L2）

```
条件:
  直近 warm_days 日分の day_summary に
  同一テーマ（キーワードグループ）が 3日以上にわたって登場している
  かつ
  そのテーマに関する personality_trends.md の「関係性の変化」セクションに
  対応するエントリが存在しない（または最終更新から 30日以上経過）
```

**実装**: day_summary のテキストから AgentCore がキーワードを抽出し、出現日数を集計する（Python の Counter を使用）。LLM は不使用。

**トリガー確認タイミング**: セッション開始時（Warm Memory ロード後）に1回評価する。セッション中は再評価しない。

#### トリガー T2: 感情の傾向変化（L5）

```
条件:
  直近 warm_days 日分の day_summary に
  ポジティブな感情表現（「嬉しい」「楽しかった」「打ち解けた」等）または
  ネガティブな感情表現（「悲しかった」「寂しかった」「落ち込んだ」等）が
  過半数の日（warm_days / 2 以上）に登場している
  かつ
  personality_trends.md の「感情の傾向」セクションに
  対応するエントリが存在しない（または最終更新から 30日以上経過）
```

**実装**: day_summary のテキストに対して感情キーワードリストでマッチングする（ルールベース）。

**感情キーワードリスト（初期定義）**:

```python
POSITIVE_EMOTION_KEYWORDS = [
    "嬉しい", "嬉しかった", "楽しい", "楽しかった", "よかった",
    "打ち解け", "仲良く", "ほっとした", "安心", "嬉しそう"
]

NEGATIVE_EMOTION_KEYWORDS = [
    "悲しい", "悲しかった", "寂しい", "寂しかった", "落ち込んだ",
    "辛かった", "怒った", "不安", "心配", "つらかった"
]
```

#### トリガー T3: 新しい口癖候補（supplementary_styles）

```
条件:
  直近 14日間の observations に
  同一のフレーズ（3〜15文字）が mascot の発言として 5回以上登場している
  かつ
  そのフレーズが persona_core.md の C7（口癖リスト）に含まれていない
  かつ
  personality_trends.md の「新しい口癖候補」セクションに
  そのフレーズが存在しない
```

**実装**: observations テーブルに対して SQL クエリで高頻度フレーズを抽出する（FTS5 を活用）。LLM は不使用。

**パフォーマンス見積もり**: 1日50件 × 14日 = 700件の短文 n-gram 分析は数十ミリ秒以内。NFR-5（起動10秒以内）への影響は無視可能。

```sql
-- 直近 14日間の mascot 発言から高頻度フレーズを抽出するクエリ（概念）
SELECT content FROM observations
WHERE speaker = 'mascot'
  AND created_at > unixepoch('now', '-14 days')
ORDER BY created_at DESC;
-- Python 側で n-gram 分析を行い、3〜15 文字のフレーズの出現頻度を集計
```

#### トリガー評価の優先順位

```
T1（関係性の変化） > T2（感情の傾向変化） > T3（口癖候補）
```

同一セッション内で複数のトリガーが満たされても、優先順位の高いものを1件のみ提案する。ただし T1 と T2 が同時に満たされており、かつ前回提案から 7日以上経過している場合は例外として両方を提案可能とする。

### 5.2 提案形式（システムプロンプト内の指示）

AgentCore がトリガー条件を検出した場合、システムプロンプトの `[S7] 応答規範ブロック` に以下の追加指示を一時的に挿入する（当該ターンのみ有効）。

#### T1（関係性の変化）のプロンプト追加指示

```
【今ターンの追加指示】
あなたとユーザーの最近の会話から、関係性に関する傾向が読み取れます。
今回の応答の最後（または会話の自然な流れの中）で、以下のフォーマットに従い
personality_trends への記録提案を1件行うこと:

---personality_trends_proposal---
種別: 関係性の変化
内容: [具体的な傾向を1-2文で記述。例: 最近 {{detected_theme}} の話題で盛り上がることが多い]
---proposal_end---

提案は自然な会話の流れの中で行い、「記録に残してもいいですか？」等のキャラクターらしい言葉を添えること。
```

#### T2（感情の傾向変化）のプロンプト追加指示

```
【今ターンの追加指示】
あなたとユーザーの最近の会話から、感情的な傾向が読み取れます。
今回の応答の最後（または会話の自然な流れの中）で、以下のフォーマットに従い
personality_trends への記録提案を1件行うこと:

---personality_trends_proposal---
種別: 感情の傾向
内容: [感情傾向を1-2文で記述。例: 最近 {{detected_emotion_direction}} な感じの会話が続いている]
---proposal_end---

提案はキャラクターの口調に合わせた自然な言葉で行うこと。
```

#### T3（口癖候補）のプロンプト追加指示

```
【今ターンの追加指示】
あなたの発言の中で「{{detected_phrase}}」というフレーズを最近よく使っていることに気づきました。
今回の応答の中で、このフレーズを口癖候補として記録提案すること:

---personality_trends_proposal---
種別: 口癖候補
内容: 「{{detected_phrase}}」をよく使うようになった
---proposal_end---

自然な会話の流れで提案すること。
```

### 5.3 承認判定ロジック

提案後、次のユーザー発言を受け取った時に AgentCore が承認/却下を判定する。

#### 承認パターン（肯定）

```python
APPROVAL_PATTERNS = [
    r"^承認$",
    r"^承認します$",
    r"^承認する$",
    r"^はい$",
    r"^うん$",
    r"^いいよ$",
    r"^いいね$",
    r"^OK$",
    r"^ok$",
    r"^そうして$",
    r"^記録して$",
    r"^記録しといて$",
    r"^残して$",
    r"承認.*します",
    r"それ.*いいよ",
    r"それ.*記録",
]
```

**注意**: パターンマッチは行頭から行うのではなく、発言全体に対して行う。ただし否定修飾（「承認できない」「承認しない」「いいよ（皮肉）」）の誤検知を防ぐため、以下の否定パターンが含まれる場合は承認と判定しない。

```python
APPROVAL_NEGATIVE_MODIFIERS = [
    r"できない",
    r"しない",
    r"いらない",
    r"要らない",
    r"やめ",
    r"却下",
    r"ちがう",
    r"違う",
]
```

#### 却下パターン（否定）

```python
REJECTION_PATTERNS = [
    r"^いや$",
    r"^いいや$",
    r"^いやだ$",
    r"^却下$",
    r"^やめて$",
    r"^いらない$",
    r"^要らない$",
    r"^やめ$",
    r"それはいい",    # 「それはいいよ」は曖昧だが「それはいい（いらない）」の慣用句も考慮
    r"記録.*しないで",
    r"覚えなくて.*いい",
]
```

#### 判定不明（スルー）

承認・却下いずれのパターンにも一致しない場合は「判定保留」とし、提案は失効する（次回トリガー条件が満たされるまで再提案しない）。保留タイムアウトは「提案から 3ターン以上ユーザーが応答しない or 3ターン以内に他の話題に移った場合」とする。

### 5.4 personality_trends.md への追記処理

ユーザーの承認が確定した場合、AgentCore は以下のフォーマットで personality_trends.md に追記する。

#### 追記フォーマット（関係性の変化）

```markdown

### [YYYY-MM-DD] 関係性の変化

[LLM が生成した提案テキスト（---personality_trends_proposal--- の内容部分）]
```

#### 追記フォーマット（感情の傾向）

```markdown

### [YYYY-MM-DD] 感情の傾向

[LLM が生成した提案テキスト]
```

#### 追記フォーマット（口癖候補）

```markdown

### [YYYY-MM-DD] 口癖候補

- [フレーズ]
```

#### 提案テキストのパースフォールバック

AgentCore が LLM の応答から `---personality_trends_proposal---` ... `---proposal_end---` を抽出する際のフォールバック仕様:

1. **デリミタが見つからない場合**: 提案をスキップし、WARNING ログに記録する
2. **デリミタの前後に空白・改行がある場合**: strip して処理する
3. **パース失敗時**: 次回のトリガー条件充足時に再提案する

#### 追記処理の実装

追記は対応するセクション（`## 関係性の変化`、`## 感情の傾向`、`## 新しい口癖候補`）の末尾に行う。セクションが存在しない場合は新規作成する。

```
追記手順:
1. personality_trends.md を読み込む
2. 対象セクションのヘッダー（例: `## 関係性の変化`）を検索する
3. 次のセクションヘッダー（## で始まる行）の直前に追記する
4. ファイルに書き込む（書き込み失敗時は 5回リトライ、最終失敗時はエラーログ + ユーザーへの通知）
```

追記後、AgentCore は personality_trends.md を再読み込みし、次回の Hot Memory ロード時に反映されるようにする。

### 5.5 提案履歴の記録

personality_trends.md の `## 提案履歴` セクションに、承認・却下・保留のいずれについても記録する。

```markdown
## 提案履歴

- [2026-03-20] 関係性の変化: 「最近Pythonの話で盛り上がることが多い」→ 承認
- [2026-03-22] 感情の傾向: 「最近少し打ち解けてきた」→ 却下
- [2026-03-25] 口癖候補: 「ふむふむ」→ 保留（タイムアウト）
```

### 5.6 Phase 1 における制約の整理

| 制約 | 内容 |
|------|------|
| 追加 API コールなし | 提案は通常対話応答内で行う。提案生成のための単独 API コールは不可 |
| ボタン UI なし | Phase 3 まで実装しない |
| 提案頻度の制限 | 同一セッション内では最大 2件（T1+T2 の例外を除き 1件）に制限する |
| 提案間隔の制限 | 同一種別の提案は最終承認・却下から 30日以上経過しないと再提案しない |

---

### 5.7 Phase 3 への移行パス

Phase 3 で「承認制 UI」を実装する際には、以下の変更を行う。

1. 提案は通常対話応答内ではなく、専用の提案ダイアログで表示する
2. 承認・却下ボタンを UI に追加する
3. キーワードマッチによる承認判定ロジックは不要になる（UI ボタンのクリックが承認イベントになる）
4. 提案テキストの生成ロジック（システムプロンプト内の追加指示 → LLM 生成）は Phase 3 でも継続利用可能

**移行影響**: AgentCore の提案生成ロジックは変更不要。UI 層とイベントハンドラのみ変更する。

---

## 6. 影響範囲

| 影響先 | 内容 |
|--------|------|
| AgentCore | トリガー評価ロジック、提案用プロンプト追加指示の動的挿入、承認判定ロジック、追記処理 |
| D-3（プロンプトテンプレート） | `[S7] 応答規範ブロック` への動的追加指示の挿入機構（本設計で定義した追加指示フォーマットを D-3 の実装で受け入れる必要がある） |
| personality_trends.md | 追記処理の対象ファイル（セクション構造の維持が必要）|
| MemoryLoader | 承認後の personality_trends.md 再読み込みトリガー |
| observations テーブル | T3 のトリガー評価で高頻度フレーズ抽出に使用する |
| day_summary テーブル | T1・T2 のトリガー評価で Warm Memory の内容を再利用する |
| Phase 3 | 承認制 UI への移行時に AgentCore のトリガー・生成ロジックを維持したまま UI 層のみ変更する |
