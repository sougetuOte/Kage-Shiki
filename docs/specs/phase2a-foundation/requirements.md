## 1. 概要

影式 Phase 2a（基盤強化）は、Phase 1 MVP の実動作テストで発覚した7件のホットフィックス（`docs/memos/hotfix-runtime-bugs-phase1.md` 参照）を教訓として、**テスト基盤・アーキテクチャ基盤・機能基盤の3領域を強化する**。

Phase 2b（自律性コア: DesireWorker, AgenticSearch）および Phase 2c（可変層運用）への入口を安全に開くことが最終目的である。

Phase 1 MVP との差分:

| 領域 | Phase 1 MVP | Phase 2a 追加・変更 |
|------|------------|-------------------|
| テスト基盤 | 単体テスト 649件、97% カバレッジ | スモークテスト手順書、GUI 手動テストチェックリスト、統合テスト3種追加 |
| アーキテクチャ | Anthropic SDK 直結 | LLMProtocol（インターフェース）追加、既存実装は不変 |
| 機能: 対話エンジン | トランケート未実装（FR-6.2 は受入条件のみ） | トランケート優先順位アルゴリズム実装 |
| 機能: ウィザード | CLI/API レベルのみ | tkinter GUI 統合（方式選択 UI + プレビュー UI + 凍結確認 UI） |
| 機能: 応答タイミング | バックエンドのみテスト | GUI 統合後の 5 秒以内応答をテストで検証 |

## 4. 変更データモデル

### 4.1 Phase 2a で追加・変更するエンティティ

Phase 1 のデータモデル（requirements.md Section 4）は変更なし。追加要素のみを記載する。

| エンティティ | 変更種別 | 内容 |
|------------|---------|------|
| `LLMProtocol` | 新規 | `src/kage_shiki/agent/llm_client.py` に追加する typing.Protocol クラス |
| プロンプトコンテキスト（`PromptContext`） | 変更 | トランケートされた各メモリ層の文字数を管理するフィールドを追加 |
| スモークテスト手順書 | 新規ドキュメント | `docs/testing/smoke-test.md` |
| GUI 手動テストチェックリスト | 新規ドキュメント | `docs/testing/gui-manual-test.md` |

### 4.2 LLMProtocol インターフェース定義（要件レベル）

`LLMProtocol` は以下のメソッドを持つ typing.Protocol として定義する。シグネチャの具体的な型注釈は設計フェーズで決定する（D-17）。

| メソッド | 役割 | Phase 2a での実装 |
|---------|------|-----------------|
| `chat(messages, system, model, max_tokens, temperature)` | LLM に対話リクエストを送信し、応答テキストを返す | AnthropicClient が実装（既存実装をそのまま維持） |

`AgentCore`、`Wizard` など上位モジュールが参照する型を `AnthropicClient` から `LLMProtocol` に変更する。

### 4.3 トランケートアルゴリズム仕様（要件レベル）

コンテキストウィンドウ超過時に適用する削除優先順位（低い数字を先に削除）:

| 優先度 | レイヤー | 説明 |
|--------|---------|------|
| 1（最初に削除） | Cold Memory（`observations` FTS5 検索結果） | 汎用的な過去記憶 |
| 2 | Warm Memory（`day_summary` 直近分） | 日次サマリー（古い順に削除） |
| 3 | Session Context（現セッションの会話ターン） | 現セッション（古いターンを削除） |
| 4（最後まで残す） | Hot Memory（`persona_core.md` 等） | 人格・ユーザー情報 |

この優先順位は Phase 1 `requirements.md` の FR-6.2 受入条件（「先に削除されるのは Cold」）と整合する。

### 5.2 ウィザード GUI 画面フロー（追加）

### 10.2 [Affirmative]（推進者）

- Phase 1 ホットフィックス 7 件の教訓が要件に直接反映されており、同じ問題を繰り返さない設計
- 統合テスト3種（B-7/B-8/B-9）は単体テストで発見できなかった Bug-2/Bug-4/Bug-5 に対応する的確な要件
- LLMProtocol の限定スコープ（Protocol 定義のみ、実装変更なし）はリスクを最小化しながら Phase 3 への拡張パスを確保する
- トランケート優先順位（人格 > セッション > Warm > Cold）は Phase 1 FR-6.2 の受入条件と整合し、ユーザーが最も感じる「人格維持」を優先する合理的な設計
- ウィザード GUI 統合は「初回起動で GUI 外の操作が不要」というユーザー体験の完結に不可欠
- 全要件の受入条件が観察可能・測定可能な形式で記述されており、テスト実装に直結する

### 10.3 [Critical]（批判者）

1. **統合テストのフレイキネス**: FR-8.3〜8.5 は実スレッドを使用するため、タイミング依存のフレイキーなテストになるリスクがある。`sleep()` やレース条件が CI を不安定にする可能性
2. **ウィザード GUI のスコープ膨張**: FR-8.8〜8.10 の「既存 wizard.py のビジネスロジックを再利用」という制約（C-4）が実際に守れるか不明。tkinter との統合で想定外のリファクタリングが発生する可能性がある
3. **LLMProtocol の静的型チェック**: `typing.Protocol` は構造的サブタイピングを使用するため、`AnthropicClient` が型チェックで互換性を持つことの検証が設計時に必要（D-17 に委譲済みだが、シグネチャ不一致のリスク）
4. **応答タイミング 5 秒の根拠**: FR-8.11 の「5 秒以内」はモックを使用したテストのため、実際の LLM 呼び出しでの応答速度保証にはならない。テストの意図と限界を明示すべき
5. **トランケート単位の曖昧さ**: D-18 で「文字数で近似」としているが、Anthropic API のトークン制限は実トークン数。長い日本語は文字数とトークン数が大きく乖離する場合がある

### 10.4 [Mediator]（調停者）

1. **統合テストのフレイキネス（対応）**: FR-8.3 の受入条件に「実スレッド」を明記しつつ、テスト設計では `threading.Event` や `queue.join()` でスレッド完了を明示的に待つことを D-20 で設計フェーズに委譲する。`pytest-timeout` の採用可否も D-20 に含める
2. **ウィザード GUI のスコープ（対応）**: 制約 C-4「既存 wizard.py のビジネスロジックを再利用する」を明示し、GUI 層（ウィンドウ・イベント処理）の追加に限定する。wizard.py のビジネスロジックに変更が必要と判明した場合は別タスクとして切り出す
3. **LLMProtocol 型チェック（対応）**: FR-8.6 の受入条件に「mypy または pyright でエラーなし」を含めた。シグネチャの詳細は D-17 で設計時に確定させる
4. **応答タイミング 5 秒の限界（対応）**: FR-8.11 の受入条件に「LLM は LLMProtocol を実装したモックを使用する（ネットワーク不要）」と明記済み。実際のネットワーク依存での応答速度は NFR-6 のコメント「Haiku: 1-3秒・ネットワーク依存」として分離して記述する。モックテストは「GUI 統合後のキュー・スレッド間通信の遅延が 5 秒を超えないこと」を検証する目的として明確化されている
5. **トランケート単位（対応）**: D-18 に「Phase 2a では文字数（`len(text)`）で近似し、Phase 4 で精密化」と明記。日本語の文字数/トークン数乖離（おおよそ 1 文字 = 1.5-2 トークン）を考慮し、安全マージンとして上限の 80% を実運用しきい値とすることを設計フェーズ（D-18）で検討する
