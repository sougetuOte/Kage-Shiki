## Critical

### [C-1] `_parse_gitleaks_json` が parse 失敗時に空リストを返す（Silent Failure）[SE]

**ファイル**: `gitleaks_scanner.py:234`
**観点**: ソースコード品質 + セキュリティ

```python
except (json.JSONDecodeError, OSError) as exc:
    logger.error("gitleaks report parse failed: %s", exc)
    return []  # ← スキャン失敗を「シークレットなし」と誤判定
```

gitleaks は実行されたがレポートが壊れている/読めない場合、呼び出し元は「シークレット検出なし」と判断する。`code-quality-guideline.md` の「Error Swallowing / Silent Failure」に該当。

**修正案**: parse 失敗時に `scan-failed` Issue を返す。

## Warning

### [W-1] `run_protect_staged()` が未インストール時に完全に沈黙する [SE]

**ファイル**: `gitleaks_scanner.py:146-147`
**観点**: アーキテクチャ整合性

`run_protect_staged()` は未インストール時に空リストを返すのみで、logger 出力もない。ship.md の外側で呼ばれた場合、スキャン未実施が検知不能。FR-4 の「沈黙しない」設計思想に反する。

**修正案**: `logger.warning("gitleaks not installed: skipping protect scan")` を追加。

### [W-2] `except Exception` がシステム情報を Issue.message に格納する [SE]

**ファイル**: `gitleaks_scanner.py:204-213`
**観点**: セキュリティ

`exc` の内容がそのまま Issue.message に入り、summary.md に永続化される。パス情報・OS エラー詳細が漏洩する経路。

**修正案**: Issue.message を固定文字列に、exc 詳細は logger.error のみに残す。

### [W-3] `_parse_gitleaks_json` に JSON 型バリデーションがない [SE]

**ファイル**: `gitleaks_scanner.py:231`
**観点**: セキュリティ

`json.loads()` 後の `for finding in data` がリスト前提。gitleaks が予期外の JSON を出力した場合 TypeError が未ハンドル。

**修正案**: `if not isinstance(data, list):` ガード追加。

### [W-4] `_TEST_CMD_PATTERN` に vitest/cargo test がない [SE]
