# スモークテスト手順書

**作成日**: 2026-03-06
**対応 FR**: FR-8.1
**関連タスク**: T-27
**対象バージョン**: Phase 2a 以降

---

## 概要

このドキュメントは、影式 (Kage-Shiki) の Phase 完了前に実施する実動作スモークテストの手順を定義する。
97% テストカバレッジがあっても実動作で不具合が発生した Phase 1 の教訓（`docs/memos/hotfix-runtime-bugs-phase1.md`）を踏まえ、
自動テスト通過後に必ずこの手順書に従った実動作確認を行うこと。

---

## 前提条件

- Python 3.12 以上がインストールされていること
- 依存パッケージがインストールされていること（`pip install -e .`）
- `ANTHROPIC_API_KEY` が発行済みであること

---

## セクション 1: 起動手順

### 1-1. `config.toml` の準備

- [ ] `~/.kage_shiki/` ディレクトリが存在しない場合、初回起動で自動生成されることを確認する
  - **手順**: `python -m kage_shiki` を初回実行する前に `~/.kage_shiki/` の存在を確認する
  - **期待結果**: ディレクトリが存在しない場合、初回起動後に `~/.kage_shiki/config.toml` が生成される

- [ ] `~/.kage_shiki/config.toml` の内容を確認する
  - **手順**: 生成された `config.toml` を任意のテキストエディタで開く
  - **期待結果**: `data_dir`、`persona_frozen`、`gui` セクション等が含まれる有効な TOML ファイルが存在する

- [ ] （必要な場合）`config.toml` の `persona_frozen` を確認する
  - **手順**: `persona_frozen = false` の場合はウィザードが起動することを想定する
  - **期待結果**: 設定値が意図した状態になっている

### 1-2. `ANTHROPIC_API_KEY` 環境変数の設定

- [ ] 環境変数または `.env` ファイルで API キーが設定されていることを確認する
  - **手順（環境変数）**: `echo $ANTHROPIC_API_KEY`（bash）または `echo %ANTHROPIC_API_KEY%`（cmd）で値が表示されることを確認する
  - **手順（.env ファイル）**: `~/.kage_shiki/.env` に `ANTHROPIC_API_KEY=sk-ant-...` が記述されていることを確認する
  - **期待結果**: API キーが設定されており、空文字でない

### 1-3. `python -m kage_shiki` の実行

- [ ] アプリケーションを起動する
  - **手順**: ターミナルで `python -m kage_shiki` を実行する
  - **期待結果**: エラーメッセージなくプロセスが起動し、10秒以内（NFR-5）にウィンドウが表示される

### 1-4. ウィンドウ表示の確認

- [ ] メインウィンドウが表示されることを確認する
  - **手順**: 起動後にデスクトップ上にウィンドウが表示されることを目視確認する
  - **期待結果**: 影式のテキストデスクトップマスコットウィンドウが枠なし透過ウィンドウとして表示される（`overrideredirect(True)` 適用済み）

- [ ] システムトレイにアイコンが表示されることを確認する
  - **手順**: タスクバーのトレイエリアを確認する
  - **期待結果**: 影式のトレイアイコンが表示されている

## セクション 2: 対話確認手順

### 2-1. テキスト入力と送信ボタンによる送信

- [ ] 送信ボタンで入力を送信できることを確認する
  - **手順**: ウィンドウの入力欄にテキスト（例: 「こんにちは」）を入力し、送信ボタンをクリックする
  - **期待結果**: 入力欄がクリアされ、処理中であることが視覚的にわかる状態になる

### 2-2. Enter キーによる送信

- [ ] Enter キーで入力を送信できることを確認する
  - **手順**: 入力欄にテキストを入力し、Enter キーを押す
  - **期待結果**: 送信ボタンと同様に送信が実行される

### 2-3. 5秒以内の応答表示

- [ ] 応答が 5 秒以内に表示されることを確認する
  - **手順**: 送信後、時計またはストップウォッチで経過時間を計測する
  - **期待結果**: 5 秒以内（NFR-6）に応答テキストがウィンドウに表示される。Haiku モデルの場合は 1-3 秒程度が目安

### 2-4. クリックイベント（突っつき）の確認

- [ ] ウィンドウをクリックしてキャラクターが反応することを確認する
  - **手順**: 送信ボタン以外のウィンドウ領域をクリックする（突っつき操作）
  - **期待結果**: キャラクターがクリックに反応し、セリフまたはリアクションが表示される。応答は 5 秒以内に表示される

---

## セクション 3: シャットダウン確認手順

### 3-1. トレイアイコンからの終了

- [ ] トレイアイコンを右クリックして「終了」を選択する
  - **手順**: タスクバーのトレイアイコンを右クリックし、コンテキストメニューの「終了」を選択する
  - **期待結果**: コンテキストメニューが表示され、「終了」が選択可能な状態になっている

### 3-2. ウィンドウ消失の確認

- [ ] 終了後にウィンドウが消えることを確認する
  - **手順**: 「終了」選択後に数秒待つ
  - **期待結果**: デスクトップ上のウィンドウが消え、トレイアイコンも消える。プロセスが残存しない（教訓 L-3: シャットダウン 1 回限り実行）

- [ ] プロセスが残存しないことを確認する
  - **手順**: タスクマネージャーで `python.exe` または `kage_shiki` プロセスが終了していることを確認する
  - **期待結果**: 関連プロセスが全て終了している

### 3-3. ログファイルの日次サマリー生成ログ確認

- [ ] ログファイルに日次サマリー生成に関するログが記録されていることを確認する
  - **手順**: `~/.kage_shiki/logs/kage_shiki.log` をテキストエディタまたは `cat` コマンドで開く
  - **期待結果**: セッション終了時の日次サマリー生成に関するログエントリが記録されている。エラーレベルのログ（`ERROR`、`CRITICAL`）がないこと

## セクション 4: 2回目起動確認手順（永続状態）

> **目的**: 教訓 L-2「2回目起動で UNIQUE 制約エラーが発生した」の再発を防止するため、同一 `data_dir` での再起動が正常に動作することを確認する。

### 4-1. 同じ `data_dir` での再起動

- [ ] セクション 3 でシャットダウン後、再度 `python -m kage_shiki` を実行する
  - **手順**: ターミナルで `python -m kage_shiki` を再実行する（同じ `~/.kage_shiki/` を使用）
  - **期待結果**: UNIQUE 制約エラー等のエラーメッセージなく正常に起動し、ウィンドウが表示される

### 4-2. エラーなしの起動確認

- [ ] 2回目起動でログにエラーがないことを確認する
  - **手順**: `~/.kage_shiki/logs/kage_shiki.log` を確認し、2回目起動分のログエントリを確認する
  - **期待結果**: `ERROR` または `CRITICAL` レベルのログエントリがない。特に `UNIQUE constraint failed` 等の SQLite エラーが含まれていない

### 4-3. 前回の記憶を引き継いだ挨拶の確認

- [ ] 2回目起動時の挨拶（セッション開始メッセージ）が 1回目と異なることを確認する
  - **手順**: 起動直後または最初の対話時のキャラクターの発言を観察する
  - **期待結果**: キャラクターが前回のセッション内容を踏まえた発言をする（例: 前回の会話に言及する、または再会を認識した挨拶をする）。1回目の初対面挨拶とは異なる内容になっている。これは `day_summary`（日次サマリー）が次のセッションに引き継がれていることを示す

### 4-4. 2回目対話後のシャットダウン確認

- [ ] 2回目起動でも正常にシャットダウンできることを確認する
  - **手順**: セクション 3 と同様の手順でシャットダウンを実施する
  - **期待結果**: セクション 3 と同様に、ウィンドウ消失・プロセス終了・ログ記録が正常に行われる

## 合否判定

全チェック項目が通過した場合のみ、Phase 完了を宣言できる。
いずれかの項目で失敗した場合は、ログを確認して原因を特定し、修正後に再テストを実施すること。

| セクション | チェック項目数 | 合否 |
|-----------|--------------|------|
| 1. 起動手順 | 4 | - |
| 2. 対話確認手順 | 4 | - |
| 3. シャットダウン確認手順 | 3 | - |
| 4. 2回目起動確認手順 | 4 | - |
| **合計** | **15** | - |
